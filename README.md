# go-musthave-diploma-tpl

Унаследовано от шаблона репозитория для индивидуального дипломного проекта курса «Go-разработчик»

## О проекте

Этот сервис реализует **систему лояльности (bonus program)** для пользователей онлайн-платформы.

### Основные функции:
- Пользователь может загрузить номер заказа для начисления бонусов.
- Баллы начисляются после проверки статуса заказа во внешней системе расчёта (accrual system).
- Начисленные баллы можно потратить на оплату новых заказов.
- Ведётся история операций по балансу: списания и зачисления.
- Поддержка JWT-аутентификации, Gzip-сжатия, Prometheus-метрик.

### Архитектурная модель
Сервис взаимодействует с двумя внешними системами:
1. **Accrual system** — система начисления баллов за заказы (внешний HTTP-сервис).
2. **Фронтенд / мобильное приложение** — конечный клиент, через который пользователь работает с баллами.

## Нюансы по принятым решениям

### 1. Использование Gin

Вместо базового `http.ServeMux` взял фреймворк `gin`, потому что:

- Требуется обработка параметризованных маршрутов (например, `/api/user/orders`)
- Необходима поддержка `text/plain` и `application/json`, тут она проще
- Упрощена работа с JWT, Gzip, ошибками (взято с предыдущих проектов)

Решение соответствует best practices для REST API на Go.

### 2. Generics

Шаблон Repository обобщён в `internal/storage/generic.go` как демонстрация generics, но не используется, так как модель простая и основана на единой таблице `balance_operations`

### 3. Хранение сумм

Суммы хранятся в **копейках (целые числа)**, а не в `float64`, чтобы избежать ошибок округления.  
Используется паттерн **"дебет-кредит"**: каждая операция — либо списание (снятие), либо зачисление (пополнение).

### 4. Статусы заказов

Статусы заказов:
- `REGISTERED` — заказ принят, но расчёт ещё не начат
- `PROCESSING` — идёт расчёт
- `INVALID` — заказ не может быть начислен
- `PROCESSED` — расчёт завершён, баллы начислены

Внутренние статусы маппятся на внешние при обмене с системой начисления.

## Архитектура

Проект использует трёхуровневую архитектуру:

- **handlers** — HTTP-интерфейс, валидация, статусы
- **service** — бизнес-логика (проверка баланса, списание, загрузка заказа)
- **storage** — работа с БД

Такой подход позволяет:

- Изолировать бизнес-правила
- Упростить тестирование
- Повторно использовать логику (например, в CLI)

## API Endpoints

| Метод | Путь | Описание |
|------|------|--------|
| POST | `/api/user/orders` | Загрузка номера заказа |
| GET  | `/api/user/orders` | Получение списка заказов с текущими статусами |
| GET  | `/api/user/balance` | Получение текущего баланса |
| POST | `/api/user/balance/withdraw` | Списание баллов на оплату заказа |
| GET  | `/api/user/withdrawals` | История списаний |

## Начало работы

1. Склонируйте репозиторий в любую подходящую директорию на вашем компьютере.
2. В корне репозитория выполните команду `go mod init <name>` (где `<name>` — адрес вашего репозитория на GitHub без префикса `https://`) для создания модуля.
3. Убедитесь, что запущена PostgreSQL-база данных.
4. Настройте переменные окружения (см. ниже).
5. Запустите приложение: `go run cmd/main.go`.

---

## Переменные окружения

| Переменная | Назначение | Пример |
|-----------|------------|--------|
| `RUN_ADDRESS` | Адрес запуска HTTP-сервера | `:8080` |
| `DATABASE_URI` | Строка подключения к PostgreSQL | `postgres://user:pass@localhost:5432/dbname` |
| `ACCURAL_SYSTEM_ADDRESS` | Адрес внешней системы начисления | `http://localhost:8081` |
| `JWT_SECRET` | Секрет для подписи JWT-токенов | `mysecretkey` |
| `LOG_LEVEL` | Уровень логирования | `debug` или `info` |
